# アーキテクチャ設計

## 1. 文書目的

本書は、`stock_ai` の初期アーキテクチャを定義するための文書である。
目的は、実装前に責務分割、データフロー、保存方針、各レイヤーの入出力を明確にし、個人開発でも継続運用しやすい構成を作ることにある。

## 2. 設計方針

- 個人開発で保守可能な単一リポジトリ構成を採用する
- まずはローカル実行前提のバッチ処理中心で設計する
- 生データ、加工データ、特徴量、学習成果物を分離する
- 未来情報リークを防ぐため、時点整合性を最優先にする
- 最初はシンプルなベースラインモデルを中心に構築する
- 後からニュース、SNS、深層学習モデルを追加できる余地を残す

## 3. システム全体像

初期構成は以下の 6 レイヤーで構成する。

1. データ取得レイヤー
2. データ保存レイヤー
3. 前処理・特徴量生成レイヤー
4. 学習・評価レイヤー
5. 推論・バックテストレイヤー
6. レポート・メタデータ管理レイヤー

データフローの基本形は以下とする。

```text
外部データソース
  -> raw data
  -> normalized data
  -> feature dataset
  -> train/validation/test split
  -> model artifact
  -> prediction / backtest result
  -> report
```

## 5. レイヤー設計

### 5.1 データ取得レイヤー

責務:
- 無料データソースを優先して株価、財務、マクロデータを取得する
- 取得時点のメタデータを保存する
- 元データをできるだけそのまま保持する

入出力:
- 入力: API、CSV、公開データ
- 出力: `data/raw/`

設計方針:
- データソースごとに取得モジュールを分離する
- 取得処理は再実行可能にする
- ソース名、取得日時、対象期間、利用条件を記録する

### 5.2 データ保存レイヤー

責務:
- 生データと加工データを明確に分離する
- 日付、銘柄コード、指標名などのキーを正規化する
- データセット版管理の基盤を作る

保存方針:
- `data/raw/`: 取得直後の原本
- `data/interim/`: 正規化・結合前の中間成果物
- `data/processed/`: 学習・推論に使う確定済みデータ
- `data/metadata/`: 取得履歴、データ辞書、版情報

推奨形式:
- 中間・加工データは Parquet を優先する
- 小規模な設定系メタデータは JSON または YAML を使う

### 5.3 前処理・特徴量生成レイヤー

責務:
- データの正規化、欠損処理、時系列整列
- ラベル生成
- 特徴量生成
- 特徴量セット定義の管理

主要処理:
- 銘柄コード統一
- 営業日カレンダーへの整列
- 修正株価ベースの騰落率計算
- 約 60 営業日先ラベルの作成
- テクニカル、財務、マクロ特徴量の作成

注意点:
- 特徴量計算時に未来データを参照しない
- 財務データは公表タイミングを意識したマージが必要
- マクロデータは公表ラグの扱いを明示する

### 5.4 学習・評価レイヤー

責務:
- 学習用データセット作成
- 時系列分割
- モデル学習
- オフライン評価
- 実験条件の保存

初期モデル候補:
- Logistic Regression
- LightGBM
- XGBoost

初期方針:
- 最初の主力は木系モデルと線形モデルの比較にする
- ニューラルモデルは後段の拡張とする

出力:
- `models/` にモデル成果物
- `reports/` に評価結果
- `data/metadata/` に実験条件

### 5.5 推論・バックテストレイヤー

責務:
- 任意基準日時点で推論を実行する
- 上位スコア銘柄を抽出する
- 簡易売買ルールでバックテストする

推論要件:
- 基準日時点で利用可能な特徴量のみを使用する
- 予測確率、スコア、主要特徴量を出力する

バックテスト要件:
- 上位 N 銘柄選定
- 定期リバランス
- 約 60 営業日保有
- 手数料とスリッページをパラメータ管理

### 5.6 レポート・メタデータ管理レイヤー

責務:
- 学習結果、評価結果、バックテスト結果を保存する
- 実験条件とモデル版の追跡を可能にする
- 将来の比較検証を容易にする

初期方針:
- 初期段階では MLflow を必須にせず、ローカルの JSON / YAML / CSV で管理する
- 管理が煩雑になったら MLflow 導入を検討する

## 6. アプリケーション構成

### 6.1 モジュール責務

- `src/stock_ai/data/`
  - 取得、読み込み、正規化
- `src/stock_ai/features/`
  - 特徴量生成、ラベル生成
- `src/stock_ai/modeling/`
  - 学習、保存、読み込み
- `src/stock_ai/evaluation/`
  - 指標計算、時系列分割、リーク検査
- `src/stock_ai/backtest/`
  - 売買ルール、損益計算、ベンチマーク比較
- `src/stock_ai/inference/`
  - 推論パイプライン、出力整形
- `src/stock_ai/reporting/`
  - レポート生成
- `src/stock_ai/utils/`
  - 共通処理

### 6.2 実行エントリポイント

初期段階では CLI ベースを想定する。

候補コマンド:
- `python -m stock_ai.data.fetch_prices`
- `python -m stock_ai.data.fetch_fundamentals`
- `python -m stock_ai.features.build_dataset`
- `python -m stock_ai.modeling.train`
- `python -m stock_ai.inference.predict`
- `python -m stock_ai.backtest.run`

## 7. データ契約

### 7.1 基本キー

主要なデータ結合キーは以下を基準とする。

- `date`
- `ticker`
- `market`

必要に応じて以下も保持する。

- `source`
- `as_of_date`
- `published_at`
- `sector`

### 7.2 ラベルデータ

最低限、以下の列を持つ。

- `date`
- `ticker`
- `close`
- `close_t_plus_60bd`
- `future_return_60bd`
- `label`

### 7.3 特徴量データ

最低限、以下を持つ。

- 識別列: `date`, `ticker`
- 学習対象外メタ列: `market`, `sector`
- 数値特徴量列
- ラベル列

## 8. 設定管理

設定はコード直書きを避け、`configs/` に集約する。

想定設定:
- `configs/data/`
  - データソース設定
  - 対象期間
  - 対象ユニバース
- `configs/features/`
  - 利用特徴量
  - ルックバック期間
  - 欠損処理
- `configs/train/`
  - モデル種別
  - ハイパーパラメータ
  - 学習期間
- `configs/inference/`
  - 推論対象日
  - 出力先
- `configs/backtest/`
  - 売買ルール
  - 保有期間
  - 手数料

## 9. 実験管理

最低限、以下を保存対象とする。

- 使用データセット版
- 使用特徴量セット
- 使用設定ファイル
- 学習期間と検証期間
- モデル種別とハイパーパラメータ
- 評価指標
- 生成成果物の保存先

## 10. 主要リスクと対策

### 10.1 データリーク

リスク:
- 将来の財務値や修正済み情報の混入

対策:
- `as_of_date` または `published_at` を保持する
- 学習前にリーク検査を入れる

### 10.2 データ品質不足

リスク:
- 無料データの欠損、更新遅延、項目不足

対策:
- ソースごとの品質評価を行う
- 必要時のみ有料データを追加検討する

### 10.3 過剰最適化

リスク:
- 特定期間に最適化されすぎる

対策:
- ウォークフォワード評価を標準化する
- 指標を複数持つ

## 11. 今後の拡張ポイント
