# 要件定義書

## 1. 文書概要

### 1.1 目的
個人開発で運用可能な株式予測 AI システムを構築する。
本システムは、過去の株価・企業情報・マクロ情報等の時系列データをもとに、ある銘柄が「基準日時点から約 60 営業日後に 10%以上上昇しているか」を分類することを目的とする。

### 1.2 文書の位置づけ
本書は、コード実装前の初版要件定義書である。
未確定事項は `TBD` として残し、今後の設計・実装・検証の基準とする。

### 1.3 想定読者
- 開発者本人
- 将来的に協力する可能性がある開発者
- AI エージェント

## 2. 背景と狙い

### 2.1 背景
- 個人でも継続的に検証可能な株式予測基盤を作りたい
- 単純な価格予測よりも、投資判断に近い分類問題として扱いたい
- 再学習、バックテスト、推論を同一基盤で回せるようにしたい

### 2.2 解きたい課題
- 多様な過去データを統合し、将来の上昇有無を分類する
- 過学習や情報リークを避けつつ、実運用に耐える評価方法を採用する
- 個人開発でも保守可能な構成にする

## 3. システム目標

### 3.1 プロダクト目標
- 銘柄単位で将来上昇確率を出力できること
- 学習から評価、推論までを再現可能に実行できること
- 入力特徴量、学習条件、評価結果を追跡可能であること

### 3.2 予測タスク定義
- タスク種別: 二値分類
- 予測対象: 個別株
- 判定条件: 基準日時点から約 60 営業日後の騰落率が 10%以上かどうか
- 出力:
  - `label`: 0 または 1
  - `probability`: 上昇確率
  - `score`: モデル内部スコア

### 3.3 成功条件
- 時系列分割に基づく検証で、ベースラインを上回る分類性能を示すこと
- バックテストで、閾値ベースの売買戦略がランダム選択や単純ルールを上回る可能性を示すこと
- 学習・推論パイプラインがローカル環境で再実行可能であること

## 4. スコープ

### 4.1 初期スコープ
- 過去データの収集・整形
- 学習用データセット生成
- 約 60 営業日後に 10%以上上昇するかの二値分類
- ベースラインモデル構築
- 時系列評価
- バックテストの最小実装
- 推論結果の保存

### 4.2 初期スコープ外
- 高頻度取引
- リアルタイム超低遅延推論
- 完全自動売買
- 信用取引やデリバティブ対応
- 本番インフラの大規模分散処理

## 5. 想定ユーザー

### 5.1 主要ユーザー
- 開発者本人

### 5.2 利用シナリオ
- 過去データを更新し、モデルを再学習する
- 任意時点で銘柄ごとの上昇確率を推論する
- バックテスト結果を確認し、特徴量やモデルを改善する

## 6. 業務要件・機能要件

### 6.1 データ収集
システムは以下のカテゴリのデータを扱えること。

- 株価データ: 始値、高値、安値、終値、出来高、調整後価格
- 財務データ: 売上、利益、EPS、PER、PBR 等
- 企業属性データ: 業種、時価総額、市場区分
- マクロデータ: 金利、指数、為替、コモディティ等
- ニュース・センチメントデータ: 初期スコープ外、将来拡張で対応

最低要件:
- データ取得元ごとに取得日時とソースを記録する
- 欠損、休日、上場廃止、株式分割等のイベントに配慮する
- 学習時点以降の情報を混入させない

### 6.2 データ前処理
- 日次ベースで時系列を整列できること
- 特徴量生成前に欠損処理ルールを適用できること
- 銘柄コード、日付、対象市場を正規化できること
- ラベル生成条件を設定ファイルで管理できること

### 6.3 特徴量生成
最低限、以下の特徴量群を扱えること。

- 価格系列特徴量: リターン、移動平均、ボラティリティ、出来高変化率
- 相対比較特徴量: 市場指数比、セクター比
- 財務特徴量: バリュエーション、成長率、収益性
- マクロ特徴量: 金利、為替、景況関連指標
- イベント特徴量: 決算日、配当、株式分割等
- テキスト特徴量: 初期スコープ外、将来拡張で対応

### 6.4 学習
- 複数モデルを差し替え可能な構成であること
- 初期ベースラインとして以下を候補に含めること
  - Logistic Regression
  - LightGBM または XGBoost
  - 時系列特徴量を扱う簡易ニューラルモデル `TBD`
- 学習条件、特徴量セット、ハイパーパラメータを保存できること

### 6.5 評価
- 時系列を保った train/validation/test 分割を採用すること
- 以下の指標を出力すること
  - Accuracy
  - Precision
  - Recall
  - F1
  - ROC-AUC
  - PR-AUC
  - 上位確率銘柄の平均リターン
- 銘柄横断のリークや未来情報混入を検査できること

### 6.6 バックテスト
- 予測スコア上位銘柄を定期リバランスする簡易戦略を評価できること
- 手数料、スリッページ、保有期間をパラメータ化できること
- ベンチマーク比較を行えること

### 6.7 推論
- 任意基準日時点で利用可能な情報のみを使って推論できること
- 出力結果を CSV または Parquet で保存できること
- 銘柄ごとの予測根拠として主要特徴量重要度を確認できること

## 7. 非機能要件

### 7.1 再現性
- 同一入力と設定で同一結果を再現しやすい構成であること
- データセット版、特徴量版、モデル版を識別できること

### 7.2 保守性
- データ取得、特徴量生成、学習、評価、推論を責務ごとに分離する
- AI エージェントが読めるようにドキュメントとディレクトリ規約を整備する

### 7.3 拡張性
- 新しいデータソースや特徴量を追加しやすい構成であること
- 分類以外の回帰タスクにも将来拡張可能であること

### 7.4 実行環境
- ローカル PC で実行可能であること
- GPU 利用は任意、CPU のみでも最小構成が動くこと
- OS: Linux を第一候補とし、ローカル実行可能であること
- 言語: Python を第一候補とする

## 8. 制約条件

- 個人開発で扱えるコストに収めること
- 利用データのライセンスを遵守すること
- 基本は無料データソースを優先し、性能または情報量の不足が明確な場合のみ有料データソースを許容すること
- 初期段階では、自己利用を前提とした分析・研究・検証用途を主目的とすること
- 過去データに対する過剰最適化を避けること

## 9. 想定アーキテクチャ

初版では以下の構成を想定する。

1. データ取得層
2. データ保存層
3. 特徴量生成層
4. 学習・評価層
5. 推論・バックテスト層
6. レポート出力層

保存形式候補:
- 生データ: CSV / JSON / API レスポンス保存
- 中間データ: Parquet
- モデル成果物: pickle / joblib / ONNX `TBD`
- 実験管理: 初期段階ではローカルメタデータ保存を採用し、必要に応じて MLflow を検討する

## 10. データ定義

### 10.1 学習サンプル単位
- 1 レコード = 1 銘柄 x 1 基準日

### 10.2 ラベル定義
- `future_return_60bd = (price_t_plus_60_business_days - price_t) / price_t`
- `label = 1 if future_return_60bd >= 0.10 else 0`

### 10.3 閾値
- `threshold_x`: 10%
- 保有期間定義: 約 60 営業日

### 10.4 対象ユニバース
- 対象市場: 日本株
- 候補例: 東証プライム、東証スタンダード、東証グロースを含む全体または一部
- 流動性フィルタ: `TBD`

## 11. 評価方針

### 11.1 オフライン評価
- 時系列ウォークフォワードを原則とする
- 期間ごとの性能変動を追跡する
- クラス不均衡を考慮する

### 11.2 投資観点の評価
- 上位 N 銘柄選定時の平均リターン
- 最大ドローダウン
- シャープレシオ相当指標
- 売買回転率

## 12. リスク

- データライセンス上の制約
- データ欠損や修正株価処理の不備
- 決算発表日時などの厳密な時点管理不足
- 過剰最適化
- 市場レジーム変化による性能劣化
- ニュース等の非構造化データの品質ばらつき

## 13. 初期マイルストーン

### Phase 1
- 対象市場とラベル閾値を確定する

- ベースライン分類モデルを構築する

### Phase 2

### Phase 3

- モデル改善と運用フロー整理を行う
